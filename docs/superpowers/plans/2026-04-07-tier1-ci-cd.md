# Tier 1 CI/CD Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Wire up GitHub Actions to build/push the `snip` image to DockerHub and automate Replicated release creation, testing, and Unstable promotion.

**Architecture:** Two workflow files (`pr.yml` and `release.yml`) each build the image tagged with the Git SHA, update `values.yaml` with that tag, create a Replicated release from `chart/snip`, and run a compatibility test. The release workflow additionally promotes to `Unstable`. A `.replicated` file at the repo root identifies the app slug.

**Tech Stack:** GitHub Actions, `docker/build-push-action`, `replicatedhq/replicated-actions`, `yq` for YAML mutation, DockerHub private registry.

## Task 3: Create PR workflow

**Files:**
- Create: `.github/workflows/pr.yml`

On every pull request to `main`: build the image, push to DockerHub tagged with the Git SHA, inject that SHA as `image.tag` in the chart's values, create a Replicated release, and run the compatibility matrix.

- [ ] **Step 1: Create directory**

```bash
mkdir -p .github/workflows
```

- [ ] **Step 2: Create .github/workflows/pr.yml**

```yaml
name: PR

on:
  pull_request:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and push image
        uses: docker/build-push-action@v6
        with:
          push: true
          tags: ajpio/snip:${{ github.sha }}
          platforms: linux/amd64

      - name: Set image tag in chart values
        uses: mikefarah/yq@v4
        with:
          cmd: yq -i '.image.tag = "${{ github.sha }}"' chart/snip/values.yaml

      - name: Create Replicated release
        uses: replicatedhq/replicated-actions/create-release@v1
        id: create-release
        with:
          app-slug: snip-enterprise
          api-token: ${{ secrets.REPLICATED_API_TOKEN }}
          chart: chart/snip

      - name: Run compatibility matrix
        uses: replicatedhq/replicated-actions/compatibility-matrix@v1
        with:
          app-slug: snip-enterprise
          api-token: ${{ secrets.REPLICATED_API_TOKEN }}
          kubernetes-target-version-range: ">=1.28.0"
          releases-to-test: ${{ steps.create-release.outputs.release-sequence }}
```

- [ ] **Step 3: Commit**

```bash
git add .github/workflows/pr.yml
git commit -m "ci: add PR workflow — build, release, compatibility test"
```

## Task 4: Create release workflow

**Files:**
- Create: `.github/workflows/release.yml`

On merge to `main`: same build/push steps (plus a `latest` tag), create release, run compatibility test, then promote to `Unstable`.

- [ ] **Step 1: Create .github/workflows/release.yml**

```yaml
name: Release

on:
  push:
    branches: [main]

jobs:
  release:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Log in to DockerHub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Build and push image
        uses: docker/build-push-action@v6
        with:
          push: true
          tags: |
            ajpio/snip:${{ github.sha }}
            ajpio/snip:latest
          platforms: linux/amd64

      - name: Set image tag in chart values
        uses: mikefarah/yq@v4
        with:
          cmd: yq -i '.image.tag = "${{ github.sha }}"' chart/snip/values.yaml

      - name: Create Replicated release
        uses: replicatedhq/replicated-actions/create-release@v1
        id: create-release
        with:
          app-slug: snip-enterprise
          api-token: ${{ secrets.REPLICATED_API_TOKEN }}
          chart: chart/snip

      - name: Run compatibility matrix
        uses: replicatedhq/replicated-actions/compatibility-matrix@v1
        with:
          app-slug: snip-enterprise
          api-token: ${{ secrets.REPLICATED_API_TOKEN }}
          kubernetes-target-version-range: ">=1.28.0"
          releases-to-test: ${{ steps.create-release.outputs.release-sequence }}

      - name: Promote to Unstable
        uses: replicatedhq/replicated-actions/promote-release@v1
        with:
          app-slug: snip-enterprise
          api-token: ${{ secrets.REPLICATED_API_TOKEN }}
          channel-to: Unstable
          release-sequence: ${{ steps.create-release.outputs.release-sequence }}
```

- [ ] **Step 2: Commit**

```bash
git add .github/workflows/release.yml
git commit -m "ci: add release workflow — build, release, test, promote to Unstable"
```

## Task 6: Create Replicated RBAC policy for CI

**Files:** None (Vendor Portal UI steps)

Create a scoped RBAC policy so the CI token cannot make destructive changes to customers, billing, or team settings.

- [ ] **Step 1: Open Vendor Portal → Team → RBAC**

Navigate to: `vendor.replicated.com` → Team → RBAC

- [ ] **Step 2: Create a new policy named `ci-policy`**

Set the policy definition to deny everything except what CI needs. Use this JSON policy body:

```json
{
  "v1": {
    "name": "ci-policy",
    "resources": {
      "allowed": [
        "kots/app/[:appId]/release/create",
        "kots/app/[:appId]/release/promote",
        "kots/app/[:appId]/release/list",
        "kots/app/[:appId]/release/read",
        "kots/app/[:appId]/channel/list",
        "kots/app/[:appId]/channel/read"
      ],
      "denied": [
        "kots/**"
      ]
    }
  }
}
```

Note: `allowed` takes precedence over `denied` — the explicit allows override the wildcard deny.

- [ ] **Step 3: Create a Service Account and assign ci-policy**

In Vendor Portal → Team → Service Accounts:
1. Create a new service account named `ci`
2. Assign the `ci-policy` RBAC policy to it
3. Generate a token for the service account
4. Update the `REPLICATED_API_TOKEN` GitHub secret with this scoped token (not your personal API token)
